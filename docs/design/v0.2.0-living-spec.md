This design satisfies the architect SKILL's required 8 sections (Summary, Modules, Interfaces, Data flow, Errors, Dependencies, Traceability, Open questions). Two additional sections required by sub-issue #306 follow the 8: **Submodule mechanism** and **Deferred to publish follow-up**.

The contract is `feature-ship`; deferrals are explicit and named, never silent.

## 1. Summary

`safer-by-default` v0.2.0 composes the sister codemod `@chughtapan/safer-spec-development` into the plugin as a required dependency. **v0.2.0 is dogfood-only**: the supported adopter is a TS+vitest workspace at the canonical path `$SBD_ROOT/dogfood/` inside any `chughtapan/safer-by-default` clone with the submodule populated (the maintainer's own clone in practice; forks behave identically); external adopters wait for the publish follow-up (spec Non-goal already names "no npm publish"). The integration uses a git submodule rather than an npm dep: `vendor/safer-spec-development/` carries the floor-pinned sister release; `bin/safer-gen-skills` inlines wrapper-skill bodies from the submodule; the dogfood workspace's `/safer:setup` installs the codemod via `pnpm add -D link:../vendor/safer-spec-development` from the dogfood path (canonical relative path; v0.2.0 supports **pnpm only** for the dogfood — see §9.3 for rationale). The skill formerly called `/safer:spec` is renamed `/safer:contract` — hard cut, no alias — and two new wrapper skills `/safer:contract-init` and `/safer:contract-migrate` expose the codemod's per-folder bootstrap and migration flows. The codemod's typed exit codes 10/11/12/13 from `safer-spec validate` route HOLD verdicts mechanically through `/safer:verify` to the right upstream modality.

The change shape: rename one skill, add two wrappers, add a submodule, extend `bin/safer-gen-skills`, modify five existing SKILL.tmpl bodies (setup, verify, architect, implement-staff, plus a cross-reference in implement-junior and implement-senior), add a doctrine paragraph to `PRINCIPLES.md`, add 9 scenarios, and update READMEs/CHANGELOG. No new `bin/` helpers (Invariant 4); the sister-package CLI is the integration surface.

### Folder shape

Both new skill folders (`skills/contract-init/`, `skills/contract-migrate/`) are **tree-shaped peers** under `skills/`, identical in shape to the existing 16 skills. The `vendor/` folder is a tree-shaped peer at repo root holding one submodule today, scalable to N submodules without re-layout. No layer-shaped folder is introduced.

The codebase declares one shape per level. `skills/` remains tree-shaped (each modality is one bounded responsibility, peers don't depend on each other). `vendor/` is tree-shaped (each vendored repo is one bounded source). `lib/`, `lsp/`, and `scenarios/` shapes are unchanged.

## 3. Interfaces

Three new and one extended interface. Every interface is declared in its terminal form below; downstream `/safer:implement-staff` writes the bodies.

### 3.1 `skills/contract-init/SKILL.tmpl`

```yaml
---
name: contract-init
version: 0.1.0
description: |
  Wrapper that exposes the sister codemod's safer-spec-init skill inside
  safer-by-default. Body is inlined at bin/safer-gen-skills time from
  vendor/safer-spec-development/skills/safer-spec-init/SKILL.md.
  Use when an adopter wants to bootstrap the living-spec layer for a new
  per-folder MODULE.md.
triggers:
  - bootstrap module spec
  - init the living-spec
  - new module-level contract
  - seed sidecar
disable-model-invocation: true
allowed-tools:
  - Bash
  - Read
  - Write
  - Edit
  - Grep
  - Glob
  - AskUserQuestion
---

# /safer:contract-init

{{> vendor-skill:safer-spec-init}}
```

**Intent:** the wrapper file's frontmatter is **hardcoded in the wrapper**, NOT inlined from the vendor body. Per §3.3 the directive strips the vendor's leading frontmatter when inlining. The wrapper author chooses the triggers, allowed-tools, and other frontmatter fields explicitly — there is no "preserve from sister" magic. The body content (post-frontmatter) is what gets inlined. When the sister releases a new version with different content (skills/<slug>/SKILL.md body), the wrapper frontmatter stays the same unless the wrapper author edits it. The interface stub committed to this branch at `skills/contract-init/SKILL.tmpl` carries the exact frontmatter shown above. **Identical shape for `contract-migrate`** (§3.2), with the corresponding contract-migrate triggers + vendor slug.

### 3.3 `bin/safer-gen-skills` — `{{> vendor-skill:<slug>}}` directive

The existing renderer recognises `{{> principles}}` and substitutes `PRINCIPLES.md`. The extension adds one more directive arm. The interface contract:

```
Input  template directive : {{> vendor-skill:<slug>}}
Output (rendered)         : the BODY of vendor/safer-spec-development/skills/<slug>/SKILL.md
                            with its leading YAML frontmatter block stripped.
Resolution path           : $PLUGIN_DIR/vendor/safer-spec-development/skills/<slug>/SKILL.md
                            (relative to bin/safer-gen-skills's PLUGIN_DIR)
Slug constraint           : matches ^[a-z0-9-]+$ (sister-package skill-directory naming)
Missing path              : non-zero exit; STDERR carries "ERROR: vendor-skill <slug> not found at <path>"
Frontmatter handling      : skip everything from the first `---` line to the next `---` line
                            (inclusive). The wrapper file's own frontmatter is the SKILL.md's
                            frontmatter; nested frontmatter would be invalid markdown.
                            If the vendor file lacks frontmatter (the leading line is not `---`),
                            inline the entire body verbatim.
```

The `awk` arm sits next to the existing `{{> principles}}` arm; the renderer matches the directive prefix, captures the slug, opens the resolved file, and skips lines until two `---` lines have passed (or, if the first line isn't `---`, inlines from the start). The error path mirrors the existing `[ -f "$PRINCIPLES_FILE" ] || …` check.

This frontmatter-strip rule is the single most subtle piece of the vendor-inlining mechanism; the scenario `wrapper-skill-vendored-content.yaml` asserts the generated SKILL.md has exactly one frontmatter block (the wrapper's), with the vendor body content following.

### 3.4 `bin/safer-gen-skills --check` — sentinel gate

The existing `--check` flag detects stale SKILL.md files (template + principles + vendor-skills changed relative to the rendered output). The extension adds one more reason to fail under a NEW flag, `--check --release`: the literal string `__SAFER_SPEC_VERSION__` appears in any committed `SKILL.tmpl` or generated `SKILL.md`.

```
Failure mode      : sentinel-unsubstituted
Detection         : grep -nF '__SAFER_SPEC_VERSION__' skills/*/SKILL.tmpl skills/*/SKILL.md
                    (run only when --release is passed; bare --check skips this arm)
Exit              : non-zero; STDERR names every file:line that carries the literal
CI                : a new GitHub Actions workflow (.github/workflows/sentinel-substitution.yml,
                    triggered on `release/*` branch pushes and v0.2.0 release tagging) runs
                    `bin/safer-gen-skills --check --release`. Bare `--check` (run by every
                    other CI workflow + the existing stale-render check) does NOT fail on
                    the sentinel.
```

**Sentinel-timing reconciliation.** The sentinel intentionally lives in source `SKILL.tmpl` during the v0.2.0 integration PR's development (it IS the placeholder that the substitution will replace). The integration PR's FINAL commit substitutes the concrete value. After the integration PR merges to main, source is sentinel-free. `--check --release` gates that final state: if any dev-time sentinel slipped past substitution, the release CI fails. Bare `bin/safer-gen-skills --check` (the developer-loop form) skips the sentinel arm so dev branches don't trip on their own legitimately-unsubstituted source.

**What gets substituted, given the dogfood-only `link:` install.** The sentinel `__SAFER_SPEC_VERSION__` is NOT a version number in the v0.2.0 substituted state — it's the install COMMAND. Source `skills/setup/SKILL.tmpl` carries a single line like `$PM add -D @chughtapan/safer-spec-development@__SAFER_SPEC_VERSION__` during development (per spec acceptance bullet 5's literal). The integration PR substitutes that ENTIRE LINE with the `link:` mechanism block (§9.3). What the release-CI gates on is: the literal string `__SAFER_SPEC_VERSION__` does NOT appear in any committed `SKILL.tmpl` after the integration PR. The substituted form is the `link:`-protocol install whose relative path is resolved at runtime against the dogfood workspace's CWD; no templating variables in the install line. The sentinel marks "this line is the install command and must be replaced before release"; the substitution form is the architect-chosen install mechanism. Post-publish, the sentinel substitutes again to `$PM add -D @chughtapan/safer-spec-development@~X.Y.Z` (the npm-name form) and the `link:` block goes away. The sentinel is the EDIT-POINT marker, not a templating variable.

This resolves the contradiction Codex flagged: dev-mode `--check` stays green when source carries the sentinel; release-mode `--check --release` goes red if the sentinel survived to the release branch.

## 4. Data flow

Three flows describe the system. Each is one short textual walk plus an ASCII diagram.

### 4.1 Wrapper-skill rendering (build-time)

The plugin's CI and local development render `skills/contract-init/SKILL.md` and `skills/contract-migrate/SKILL.md` from their templates by inlining vendor content.

```
  skills/contract-init/SKILL.tmpl
        |
        v
  bin/safer-gen-skills
        |   reads PRINCIPLES.md (for {{> principles}} in other skills)
        |   reads vendor/safer-spec-development/skills/safer-spec-init/SKILL.md
        v
  skills/contract-init/SKILL.md        (committed; AUTO-GENERATED marker)
```

Same shape for `contract-migrate`. Error paths: missing `PRINCIPLES.md` → `MissingPrinciplesError`; missing vendor file → `MissingVendorSkillError`; stale committed `SKILL.md` under `--check` → `StaleRenderError`; sentinel present under `--check --release` → `SentinelUnsubstitutedError` (bare `--check` skips the sentinel arm per §3.4).

### 4.2 Dogfood `/safer:setup` install (runtime, in the canonical dogfood workspace `$SBD_ROOT/dogfood/`)

`/safer:setup` Step 4c installs the codemod into the dogfood workspace's `node_modules` via pnpm's `link:` protocol pointing at the submodule's working-tree path. Non-dogfood CWDs are halted at the pre-flight check.

```
  $SBD_ROOT/dogfood/  (TS + vitest workspace; canonical fixed path)
        |
        | /safer:setup invoked from this CWD
        v
  Step 4c: pre-flight halt
        |
        +---- [CWD is not exactly $SBD_ROOT/dogfood/] -> halt with
        |       "v0.2.0 is dogfood-only; run from `$SBD_ROOT/dogfood/`; external
        |        adopters wait for publish follow-up" pointer.
        |
        +---- [not inside any chughtapan/safer-by-default clone with
        |      vendor/safer-spec-development/ populated] -> halt with the
        |       "v0.2.0 is dogfood-only; init the submodule then retry" pointer.
        |
        +---- [pnpm not installed]                    -> halt with "v0.2.0 dogfood
        |                                                requires pnpm" pointer.
        |
        +---- [TS+vitest check fails]                 -> halt with "use 0.1.x" pointer.
        |
        |   [all pre-flight checks pass]
        v
  pnpm add -D "link:../vendor/safer-spec-development"
        |   (canonical relative path; lockfile entry is "link:../vendor/safer-spec-development"
        |    identical across every clone of the repo because the dogfood path is fixed)
        v
        install creates a symlink in dogfood/node_modules/ pointing at the
        vendored submodule (pnpm controls the exact node_modules layout
        depending on its hoisting mode; the package.json entry is
        "@chughtapan/safer-spec-development": "link:../vendor/safer-spec-development"
        and pnpm resolves the target correctly regardless of internal layout)
        v
  pnpm exec safer-spec doctor   ----> [failure] -> Step 11 receipt BLOCKED
        |   [doctor OK]
        v
  seed safer-spec.config.json in $SBD_ROOT/dogfood/ (never overwrite)
        |
        v
  patch vitest config in $SBD_ROOT/dogfood/ (sentinel-bounded; reporter wired)
        |
        v
  add SPEC_LAYER=required to dogfood CLAUDE.md managed section
        |
        v
  record doctor output in Step 11 receipt
```

Workspace monorepos inside `$SBD_ROOT/dogfood/` (if the dogfood layout uses a `pnpm-workspace.yaml`) repeat the seed+wire step per workspace package; the `link:` install runs once at `$SBD_ROOT/dogfood/` (the workspace root). Post-publish, the dogfood pre-flight halt and the `link:` install are both replaced by the npm-name install (sentinel substitution), and external adopters become supported.

### 4.3 `/safer:verify` ring-1 gate + exit-code routing (runtime, in the dogfood workspace)

`/safer:verify` Phase 2 probes the codemod's health; Phase 3 runs the validate gate; Phase 5 maps exit codes to HOLD targets per Principle 8.

```
  PR ready for verify
        |
        v
  Phase 2: $PM exec safer-spec doctor   ----> [failure] -> BLOCKED (verdict body: doctor output)
        |   [doctor OK]
        v
  Phase 3 ring-1: $PM exec safer-spec validate --implemented  (wrapped in 60s Node timeout)
        |
        +---- exit 0 ---> proceed (no HOLD from spec layer)
        +---- exit 10 (version skew)             ---> BLOCKED + show doctor output verbatim
        +---- exit 11 (MissingSpecPropertyError) ---> HOLD route /safer:contract
        |                                              (override if --json carries recommended_route)
        +---- exit 12 (MissingStubError)         ---> HOLD route /safer:architect
        |                                              (or /safer:implement-staff if --json names the stub)
        +---- exit 13 (MissingImplError)         ---> HOLD route /safer:implement-{junior|senior|staff}
        |                                              per --json recommended_route; fallback to
        |                                              bin/safer-diff-scope whole-PR (verdict notes
        |                                              PR-level imprecision)
        +---- timeout / hung ---------------------> BLOCKED + stderr surfaced
        +---- stale-sidecar (stop rule 7)        ---> HOLD route /safer:contract
        v
  Phase 5 verdict + Phase 6 publish (table rows for exits 10/11/12/13)
```

Exit-code routing is the mechanical surface of Principle 8 (the ratchet); no downstream modality may sidestep by editing the sidecar JSON or `@spec.*` directives directly to "make the error go away" (Invariant 2).

## 5. Errors

Every interface declares its error channel as a named tagged-error set. All errors below are discriminated by their `_tag` field and exposed via the `safer-spec validate --json` schema or the `bin/safer-gen-skills` renderer.

### 5.1 `bin/safer-gen-skills` (renderer + check)

```
type GenSkillsError =
  | { _tag: "MissingPrinciplesError";        path: string }
  | { _tag: "MissingVendorSkillError";       slug: string; resolvedPath: string }
  | { _tag: "StaleRenderError";              path: string; reason: "tmpl-newer" | "principles-newer" | "vendor-newer" }
  | { _tag: "SentinelUnsubstitutedError";    occurrences: ReadonlyArray<{ path: string; line: number }> }
```

The error channel surfaces as non-zero exit + STDERR string; `--check` returns first-error-only or all-errors per the existing `--check` semantics.

### 5.2 `safer-spec validate` (consumed by `/safer:verify` Phase 3)

The sister codemod owns these error tags; the plugin consumes them at the exit-code boundary. They are listed here for completeness — the plugin does not raise them, only routes on them.

```
type ValidateExitCode =
  | { _tag: "VersionSkewError";              code: 10; doctorOutput: string }
  | { _tag: "MissingSpecPropertyError";      code: 11; symbol: string; sidecarPath: string; recommendedRoute?: string }
  | { _tag: "MissingStubError";              code: 12; symbol: string; sidecarPath: string; recommendedRoute?: string }
  | { _tag: "MissingImplError";              code: 13; symbol: string; sidecarPath: string; recommendedRoute?: string }
```

The `recommendedRoute` field is the optional `safer-spec validate --json` override; spec Assumption 8 says its absence is acceptable for v0.2.0 with the `bin/safer-diff-scope` fallback.

### 5.3 `/safer:verify` Phase 3 timeout wrapper

```
type ValidateTimeoutError =
  | { _tag: "ValidateTimeoutError";          timeoutMs: 60000; pid: number; stderr: string }
```

The wrapper is a single-line `node -e` using `child_process.spawn` + `setTimeout`. The error surfaces as BLOCKED with the stderr verbatim in the verdict body. A unit test against a deliberately hung subprocess confirms the wrapper (spec acceptance bullet 21).

### 5.4 `/safer:setup` Step 4c

```
type SetupStep4cError =
  | { _tag: "NotDogfoodCwd";                 expectedCwd: string; actualCwd: string }
  | { _tag: "NotInSaferByDefaultClone";      walkedFrom: string }
  | { _tag: "VendorSubmoduleAbsent";         sbdRoot: string; expectedPath: string }
  | { _tag: "PnpmAbsent";                    pathSeen: ReadonlyArray<string> }
  | { _tag: "NotTypeScriptError";            reason: "no-tsconfig" | "no-vitest" }
  | { _tag: "CodemodInstallFailed";          stderr: string }
  | { _tag: "DoctorFailed";                  stderr: string }
```

Each error maps to a halt-with-pointer receipt; non-dogfood halts with the "v0.2.0 is dogfood-only; cd to $SBD_ROOT/dogfood/" pointer; non-TS halts with the "use safer-by-default 0.1.x" pointer per spec Invariant 1; vendor-absent halts with the "git submodule update --init --recursive" pointer; pnpm-absent halts with the "install pnpm" pointer. The Step 4c pre-flight in §9.3 sequences these checks so the most-actionable pointer fires first when multiple conditions fail.

The error union is exhaustive: every public function downstream `implement-*` writes will discriminate on the tag set with a `default: absurd(x)` branch (Principle 4).

### 9.1 Submodule path

**Canonical path:** `vendor/safer-spec-development/`

**Why this name:** matches spec #304 acceptance bullet 7's existing language ("a `vendor/safer-spec-development/skills/<name>/SKILL.md` snapshot ... committed to this repo"). Tree-shaped peer under `vendor/`. The folder name mirrors the sister npm package name (`@chughtapan/safer-spec-development`) so a reader scanning the repo can tell which upstream this points at without opening `.gitmodules`. `external/`, `deps/`, and `third_party/` were considered; `vendor/` wins because the spec text already says so and the `feedback_stop_over_engineering` memory rejects parallel naming.

**Setup command (run once by the implementer after S1 merges):**

```bash
git submodule add \
  https://github.com/chughtapan/safer-spec-development.git \
  vendor/safer-spec-development
cd vendor/safer-spec-development
git checkout <sister v0.2.0 release SHA>
cd ../..
git add .gitmodules vendor/safer-spec-development
git commit -m "vendor: pin safer-spec-development at v0.2.0 release SHA"
```

**Update mechanism (per sister release):**

```bash
cd vendor/safer-spec-development
git fetch && git checkout <new pinned SHA>
cd ../..
bin/safer-gen-skills            # re-render contract-init/-migrate
git add vendor/safer-spec-development skills/contract-init/SKILL.md skills/contract-migrate/SKILL.md

```

Per Non-goal 8: manual git commit on every bump. No `postinstall` hook. No CI auto-bump. No `bin/safer-vendor-skills` helper.

### 9.2 `bin/safer-gen-skills` reads wrapper bodies from the submodule

The current renderer uses awk to expand one directive (`{{> principles}}`). The extension adds a second directive (`{{> vendor-skill:<slug>}}`) that resolves to the submodule path.

**Path resolution:**

```
directive   : {{> vendor-skill:safer-spec-init}}
resolves to : $PLUGIN_DIR/vendor/safer-spec-development/skills/safer-spec-init/SKILL.md
```

Where `$PLUGIN_DIR` is already declared at the top of `bin/safer-gen-skills` as `$(cd "$(dirname "$0")/.." && pwd)`. No new env var.

**Awk template (added next to the existing `{{> principles}}` arm):**

```awk
/^\{\{> vendor-skill:[a-z0-9-]+\}\}$/ {
  # Extract slug between "vendor-skill:" and "}}"
  slug = $0
  sub(/^\{\{> vendor-skill:/, "", slug)
  sub(/\}\}$/, "", slug)
  vendor_path = vendor_root "/" slug "/SKILL.md"
  # Strip leading YAML frontmatter block (--- ... ---) per §3.3.
  # If the first line is "---", consume until the second "---" line, then inline the rest.
  # If the first line is NOT "---", inline from the start verbatim.
  in_fm = 0
  fm_closed = 0
  first_line = 1
  while ((getline line < vendor_path) > 0) {
    if (first_line) {
      first_line = 0
      if (line == "---") {
        in_fm = 1
        continue
      }
      # No frontmatter; fall through to print.
    }
    if (in_fm && !fm_closed) {
      if (line == "---") { fm_closed = 1; continue }
      continue
    }
    print line
  }
  close(vendor_path)
  next
}
```

The frontmatter-strip arm is the §3.3 specification made executable; the leading `---` triggers consumption-until-next-`---`. The wrapper's frontmatter (declared in the wrapper's own SKILL.tmpl) remains the only frontmatter in the generated `SKILL.md`.

`vendor_root` is passed in as a new `-v` flag from the bash wrapper: `-v vendor_root="$PLUGIN_DIR/vendor/safer-spec-development/skills"`. Mirrors the existing `-v principles_file="$PRINCIPLES_FILE"` flag.

**Missing path:** the awk `getline` returns 0 if the file is missing, producing empty output. To make the failure loud, the bash wrapper pre-validates every `{{> vendor-skill:<slug>}}` directive by extracting the slug and checking the file exists before rendering. Missing → exit 1 with `ERROR: vendor-skill <slug> not found at <path>`. This mirrors the existing `[ -f "$PRINCIPLES_FILE" ]` check.

**Generated SKILL.md output:** the wrapper file in the repo is one frontmatter block + one heading + one directive. ### 9.3 `/safer:setup` resolves `safer-spec` through the submodule

**Install command (in `skills/setup/SKILL.tmpl` Step 4c):**

```bash
# 1. Pre-flight: this is a dogfood-only install with a canonical fixed path.
#    The dogfood workspace MUST be exactly $SBD_ROOT/dogfood/. This fixes the
#    relative-path computation: every clone of the repo computes the same
#    "link:../vendor/safer-spec-development" lockfile entry. Nested or
#    relocated dogfood layouts are NOT supported in v0.2.0.

# Locate $SBD_ROOT by walking up from CWD looking for the safer-by-default
# clone signature. Distinguish three failure modes:
#   (i)  not inside any safer-by-default clone (no plugin.json with name=safer
#        found anywhere up the tree) → NotInSaferByDefaultClone
#   (ii) clone found but vendor/safer-spec-development/ not populated
#        → VendorSubmoduleAbsent (separate, more actionable error)
#   (iii) clone + vendor both found → SBD_ROOT set; continue.
SBD_ROOT=""
SBD_ROOT_VENDOR_ABSENT=""
cur="$PWD"
while [ "$cur" != "/" ]; do
  if [ -f "$cur/.claude-plugin/plugin.json" ] \
      && grep -q '"name": "safer"' "$cur/.claude-plugin/plugin.json" 2>/dev/null; then
    if [ -d "$cur/vendor/safer-spec-development" ]; then
      SBD_ROOT="$cur"
      break
    else
      SBD_ROOT_VENDOR_ABSENT="$cur"
      # Don't break; keep walking in case an outer clone has the vendor.
    fi
  fi
  cur=$(dirname "$cur")
done
if [ -z "$SBD_ROOT" ] && [ -n "$SBD_ROOT_VENDOR_ABSENT" ]; then
  cat >&2 <<MSG
ERROR: VendorSubmoduleAbsent

Found a chughtapan/safer-by-default clone at $SBD_ROOT_VENDOR_ABSENT, but
$SBD_ROOT_VENDOR_ABSENT/vendor/safer-spec-development/ is missing.

Initialize the submodule:
  cd $SBD_ROOT_VENDOR_ABSENT
  git submodule update --init --recursive

Then retry /safer:setup.
MSG
  exit 1
fi
if [ -z "$SBD_ROOT" ]; then
  cat >&2 <<'MSG'
ERROR: NotInSaferByDefaultClone — /safer:setup v0.2.0 is dogfood-only.

This skill runs only inside a chughtapan/safer-by-default clone with
vendor/safer-spec-development/ populated.

External adopters: stay on safer-by-default 0.1.x. v0.2.x external support
ships in the publish follow-up (npm publish of @chughtapan/safer-spec-development).
MSG
  exit 1
fi

# Enforce canonical dogfood CWD: $SBD_ROOT/dogfood/.
if [ "$PWD" != "$SBD_ROOT/dogfood" ]; then
  echo "ERROR: /safer:setup must be invoked from $SBD_ROOT/dogfood/ (canonical dogfood path)." >&2
  echo "       Current CWD: $PWD" >&2
  echo "       v0.2.0 does not support nested or relocated dogfood layouts." >&2
  exit 1
fi

# Enforce pnpm.
if ! command -v pnpm >/dev/null 2>&1; then
  echo "ERROR: /safer:setup v0.2.0 dogfood requires pnpm." >&2
  echo "       Install pnpm (https://pnpm.io) then retry." >&2
  exit 1
fi

# 2. Install via pnpm's link: protocol. The canonical relative path is fixed
#    (../vendor/safer-spec-development); the lockfile entry is identical
#    across every clone of the repo.
pnpm add -D "link:../vendor/safer-spec-development"
```

**Why pnpm only.** Codex round-4 correctly observed that `link:` semantics differ across package managers: pnpm and yarn 1/2 support `link:<path>` natively as a symlink protocol; npm has no `link:` protocol and treats `npm install <path>` as a copy (not a symlink), which breaks the maintainer-shared lockfile because the package's own internal paths drift per machine; bun's `bun add <path>` behaves like npm. For v0.2.0 dogfood, the simplest correct design is to constrain the PM to pnpm — the de-facto PM for the safer-by-default repo (per the `node_modules/` shape Claude Code installs). Yarn could be added later if dogfood broadens; bun and npm require a different install mechanism that the publish follow-up's npm-name install will cover for all PMs.

**Why a fixed canonical dogfood path (`$SBD_ROOT/dogfood/`).** Codex round-4 correctly observed that "the relative path is portable because it is repo-relative" overclaimed: the relative path depends on the dogfood CWD inside the repo. If maintainer A puts the dogfood at `$SBD_ROOT/dogfood/` and maintainer B puts it at `$SBD_ROOT/test-app/`, their `link:` entries diverge. Fixing the dogfood path to `$SBD_ROOT/dogfood/` (and rejecting other CWDs in the pre-flight) makes the lockfile entry identical across every clone.

**Why `link:` and not a tarball.** A tarball-from-submodule was the round-1/2/3 design after Codex caught the `file:` portability bug; the user's A7 adjudication (round-4) selected **dogfood-only** rather than external+tarball. With external adopters out of scope, the simpler pnpm `link:` is the right install — symlink in `node_modules`, no committed tarball, no post-publish cleanup codemod. Subpath imports (`@chughtapan/safer-spec-development/reporter` for vitest wiring) resolve through the symlink natively.

**Why `link:` and not the npm name.** Parent epic revision 2 defers npm publish; the sister package is not on the registry. Post-publish, the install swaps to `pnpm add -D @chughtapan/safer-spec-development@~X.Y.Z` via the `__SAFER_SPEC_VERSION__` substitution (spec Invariant 12); the dogfood pre-flight halt + canonical-path enforcement + pnpm-only constraint all lift at that point, since external adopters become supported via the published npm name (with full PM compatibility).

### 9.4 `/safer:verify` resolves the gate

**Phase 2 detection probe (in `skills/verify/SKILL.tmpl`):**

```bash
if ! $PM exec safer-spec doctor >/tmp/safer-verify-$PR/doctor.log 2>&1; then
  # BLOCKED — doctor failed
  VERDICT=BLOCKED
  cat /tmp/safer-verify-$PR/doctor.log  # surface verbatim in verdict body
  exit 0
fi
```

**Phase 3 ring-1 gate (added to the existing `$LINT_CMD`/`$TYPECHECK_CMD`/`$TEST_CMD` block):**

```bash
SAFER_VALIDATE_TIMEOUT_MS="${SAFER_VALIDATE_TIMEOUT_MS:-60000}"
# Disable set -e around the timed call so the non-zero exit is captured, not eaten.
set +e
TIMEOUT_MS="$SAFER_VALIDATE_TIMEOUT_MS" node -e '
  const { spawn } = require("child_process");
  // process.argv = [node, "[eval]", $PM, "exec", "safer-spec", "validate", "--implemented"]
  // (Node 18+ places the eval label at index 1; user args follow `--` from index 2.)
  const [pm, ...rest] = process.argv.slice(2);
  const timeoutMs = Number(process.env.TIMEOUT_MS) || 60000;
  const child = spawn(pm, rest, { stdio: "inherit" });
  const timer = setTimeout(() => { child.kill("SIGKILL"); process.exit(124); }, timeoutMs);
  child.on("exit", (code) => { clearTimeout(timer); process.exit(code ?? 1); });
' -- "$PM" exec safer-spec validate --implemented \
  > /tmp/safer-verify-$PR/spec-validate.log 2>&1
SPEC_VALIDATE_EXIT=$?
set -e
```

The wrapper passes `$PM` and its argv through Node's `process.argv` rather than shell-interpolating into the `-e` string. `process.argv.slice(2)` yields the user args after Node's `[eval]` label, so `pm = $PM` and `rest = ["exec", "safer-spec", "validate", "--implemented"]`; `spawn(pm, rest)` then invokes the codemod through the adopter's `$PM exec` resolution. This eliminates shell-quoting risk if `$PM` ever contains a space or metacharacter, and keeps the timeout helper a single self-contained one-liner per spec Invariant 4 (no new `bin/` helper).

The `set +e` / `set -e` bracket is required: `/safer:verify` runs under `set -e` (standard shell hygiene), and the validate command intentionally exits non-zero on exit codes 10/11/12/13. Without bracketing, the shell would exit before `SPEC_VALIDATE_EXIT=$?` runs and the routing table at Phase 5 would never fire. (Equivalent `|| SPEC_VALIDATE_EXIT=$?` form is acceptable for one-line shells; the explicit bracket is clearer for the multi-line snippet.)

Timeout default is 60s, overridable per-invocation via `SAFER_VALIDATE_TIMEOUT_MS` (env var read from the calling shell, forwarded into Node via the `TIMEOUT_MS=...` prefix).

Exit codes map per §4.3. `124` is the conventional timeout exit; verify surfaces it as BLOCKED with stderr.

`pnpm exec` resolves the codemod via the dogfood workspace's `node_modules/.bin` (the install happened during `/safer:setup`). The submodule is not directly referenced at verify time; setup's `link:` install put a symlink in `node_modules` pointing at the submodule, and `pnpm exec` finds the codemod's bin entry through that symlink. This matches spec Invariant 12 (a): local-first binary resolution via `pnpm exec`, never `command -v`. (External adopters not supported in v0.2.0 per A7; the symlink mechanism is only meaningful inside the dogfood workspace, where the submodule is initialized.)

**Why `pnpm exec` (not `$PM exec`) in verify for v0.2.0 dogfood.** Setup (§9.3) constrains the dogfood PM to pnpm; verify inherits that constraint. The actual `skills/verify/SKILL.tmpl` Phase 2/3 detection picks up `$PM` from the project's package.json `packageManager` field (which setup writes to declare pnpm) and uses it for the exec invocation. In the dogfood workspace, `$PM` always resolves to `pnpm`. Post-publish, when external adopters are supported and the `link:` install is replaced with the npm-name install, `$PM` may be any of pnpm/npm/yarn/bun and verify's invocation lifts to fully PM-agnostic.
