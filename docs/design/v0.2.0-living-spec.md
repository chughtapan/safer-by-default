**Spec:** https://github.com/chughtapan/safer-by-default/issues/304
**Parent epic:** https://github.com/chughtapan/safer-by-default/issues/305
**Sub-issue (this design):** https://github.com/chughtapan/safer-by-default/issues/306
**Doctrine SHA:** `6ce0a18` (the SHA frozen at parent-epic OK time)
**Branch:** `arch/306-v0.2.0-living-spec`

This design satisfies the architect SKILL's required 8 sections (Summary, Modules, Interfaces, Data flow, Errors, Dependencies, Traceability, Open questions). Two additional sections required by sub-issue #306 follow the 8: **Submodule mechanism** and **Deferred to publish follow-up**.

The contract is `feature-ship`; deferrals are explicit and named, never silent.

## 1. Summary

`safer-by-default` v0.2.0 composes the sister codemod `@chughtapan/safer-spec-development` into the plugin as a required dependency that adopter projects install during `/safer:setup` and that `/safer:verify` runs as a ring-1 gate. The integration uses a git submodule rather than an npm dep: `vendor/safer-spec-development/` carries the floor-pinned sister release; `bin/safer-gen-skills` inlines wrapper-skill bodies from the submodule; `/safer:setup` installs the codemod into adopter projects as `file:$CLAUDE_PLUGIN_ROOT/vendor/safer-spec-development` (deferred npm-name install is a follow-up). The skill formerly called `/safer:spec` is renamed `/safer:contract` — hard cut, no alias — and two new wrapper skills `/safer:contract-init` and `/safer:contract-migrate` expose the codemod's per-folder bootstrap and migration flows. The codemod's typed exit codes 10/11/12/13 from `safer-spec validate` route HOLD verdicts mechanically through `/safer:verify` to the right upstream modality.

The change shape: rename one skill, add two wrappers, add a submodule, extend `bin/safer-gen-skills`, modify five existing SKILL.tmpl bodies (setup, verify, architect, implement-staff, plus a cross-reference in implement-junior and implement-senior), add a doctrine paragraph to `PRINCIPLES.md`, add 9 scenarios, and update READMEs/CHANGELOG. No new `bin/` helpers (Invariant 4); the sister-package CLI is the integration surface.

### Folder shape

Both new skill folders (`skills/contract-init/`, `skills/contract-migrate/`) are **tree-shaped peers** under `skills/`, identical in shape to the existing 16 skills. The `vendor/` folder is a tree-shaped peer at repo root holding one submodule today, scalable to N submodules without re-layout. No layer-shaped folder is introduced.

The codebase declares one shape per level. `skills/` remains tree-shaped (each modality is one bounded responsibility, peers don't depend on each other). `vendor/` is tree-shaped (each vendored repo is one bounded source). `lib/`, `lsp/`, and `scenarios/` shapes are unchanged.

## 3. Interfaces

Three new and one extended interface. Every interface is declared in its terminal form below; downstream `/safer:implement-staff` writes the bodies.

### 3.1 `skills/contract-init/SKILL.tmpl`

```yaml
---
name: contract-init
version: 0.1.0
description: |
  Wrapper that exposes the sister codemod's safer-spec-init skill inside
  safer-by-default. Body is inlined at bin/safer-gen-skills time from
  vendor/safer-spec-development/skills/safer-spec-init/SKILL.md.
  Use when an adopter wants to bootstrap the living-spec layer for a new
  per-folder MODULE.md.
triggers: <preserved from sister vendor body>
allowed-tools: <preserved from sister vendor body>
---

# /safer:contract-init

{{> vendor-skill:safer-spec-init}}
```

**Intent:** the wrapper file is one frontmatter + one inline directive. `safer-gen-skills` renders the full SKILL.md by inlining the sister-vendor body verbatim. The wrapper does not duplicate or paraphrase content.

### 3.2 `skills/contract-migrate/SKILL.tmpl`

```yaml
---
name: contract-migrate

  safer-by-default. Body is inlined at bin/safer-gen-skills time from

  Use when an adopter is migrating an existing module to the living-spec

# /safer:contract-migrate

{{> vendor-skill:safer-spec-migrate}}
```

**Intent:** identical shape to `contract-init`, different vendor slug.

### 3.3 `bin/safer-gen-skills` — `{{> vendor-skill:<slug>}}` directive

The existing renderer recognises `{{> principles}}` and substitutes `PRINCIPLES.md`. The extension adds one more directive arm. The interface contract:

```
Input  template directive : {{> vendor-skill:<slug>}}
Output (rendered)         : the verbatim contents of
                            vendor/safer-spec-development/skills/<slug>/SKILL.md
Resolution path           : $PLUGIN_DIR/vendor/safer-spec-development/skills/<slug>/SKILL.md
                            (relative to bin/safer-gen-skills's PLUGIN_DIR)
Slug constraint           : matches ^[a-z0-9-]+$ (sister-package skill-directory naming)
Missing path              : non-zero exit; STDERR carries "ERROR: vendor-skill <slug> not found at <path>"
```

The `awk` arm sits next to the existing `{{> principles}}` arm; the renderer matches the directive prefix, captures the slug, and inlines the resolved file. The error path mirrors the existing `[ -f "$PRINCIPLES_FILE" ] || …` check.

### 3.4 `bin/safer-gen-skills --check` — sentinel gate

The existing `--check` flag detects stale SKILL.md files (template + principles + vendor-skills changed relative to the rendered output). The extension adds one more reason to fail: the literal string `__SAFER_SPEC_VERSION__` appears in any committed `SKILL.tmpl` or generated `SKILL.md`.

```
Failure mode      : sentinel-unsubstituted
Detection         : grep -nF '__SAFER_SPEC_VERSION__' skills/*/SKILL.tmpl skills/*/SKILL.md
                    (run after the stale check)
Exit              : non-zero; STDERR names every file:line that carries the literal
Pre-release CI    : safer-by-default's existing CI workflow runs `bin/safer-gen-skills --check`
                    BEFORE marketplace publish; release fails when sentinel is present
```

Source `SKILL.tmpl` carries the sentinel in development. The integration PR substitutes the concrete `~X.Y.Z` pin before merge. Release-time CI gates on the substituted state.

## 4. Data flow

Three flows describe the system. Each is one short textual walk plus an ASCII diagram.

### 4.1 Wrapper-skill rendering (build-time)

The plugin's CI and local development render `skills/contract-init/SKILL.md` and `skills/contract-migrate/SKILL.md` from their templates by inlining vendor content.

```
  skills/contract-init/SKILL.tmpl
        |
        v
  bin/safer-gen-skills
        |   reads PRINCIPLES.md (for {{> principles}} in other skills)
        |   reads vendor/safer-spec-development/skills/safer-spec-init/SKILL.md
        v
  skills/contract-init/SKILL.md        (committed; AUTO-GENERATED marker)
```

Same shape for `contract-migrate`. Error paths: missing `PRINCIPLES.md` → `MissingPrinciplesError`; missing vendor file → `MissingVendorSkillError`; stale committed `SKILL.md` under `--check` → `StaleRenderError`; sentinel present under `--check` → `SentinelUnsubstitutedError`.

### 4.2 Adopter `/safer:setup` install (runtime, in adopter project)

`/safer:setup` Step 4c installs the codemod into the adopter project's `node_modules` using the submodule's filesystem path.

```
  adopter project (TS + vitest)
        |
        | /safer:setup invoked
        v
  Step 4c: pre-flight TS+vitest check  ----> [non-TS] -> halt with "use 0.1.x" pointer
        |   [TS+vitest qualifies]
        v
  resolve $SAFER_SBD_PLUGIN_ROOT
        |   (env CLAUDE_PLUGIN_ROOT or ~/.claude/plugins/cache/safer-by-default/safer/<v>/)
        v
  $PM add -D "file:$SAFER_SBD_PLUGIN_ROOT/vendor/safer-spec-development"
        |
        | install lands codemod in adopter node_modules
        v
  $PM exec safer-spec doctor   ----> [failure] -> Step 11 receipt BLOCKED
        |   [doctor OK]
        v
  seed safer-spec.config.json (never overwrite)
        |
        v
  patch vitest config (sentinel-bounded; reporter wired)
        |
        v
  add SPEC_LAYER=required to project CLAUDE.md managed section
        |
        v
  record doctor output in Step 11 receipt
```

Workspace monorepos repeat the seed+wire step per workspace package; the codemod install runs once at workspace root.

### 4.3 `/safer:verify` ring-1 gate + exit-code routing (runtime, in adopter or this repo)

`/safer:verify` Phase 2 probes the codemod's health; Phase 3 runs the validate gate; Phase 5 maps exit codes to HOLD targets per Principle 8.

```
  PR ready for verify
        |
        v
  Phase 2: $PM exec safer-spec doctor   ----> [failure] -> BLOCKED (verdict body: doctor output)
        |   [doctor OK]
        v
  Phase 3 ring-1: $PM exec safer-spec validate --implemented  (wrapped in 60s Node timeout)
        |
        +---- exit 0 ---> proceed (no HOLD from spec layer)
        +---- exit 10 (version skew)             ---> BLOCKED + show doctor output verbatim
        +---- exit 11 (MissingSpecPropertyError) ---> HOLD route /safer:contract
        |                                              (override if --json carries recommended_route)
        +---- exit 12 (MissingStubError)         ---> HOLD route /safer:architect
        |                                              (or /safer:implement-staff if --json names the stub)
        +---- exit 13 (MissingImplError)         ---> HOLD route /safer:implement-{junior|senior|staff}
        |                                              per --json recommended_route; fallback to
        |                                              bin/safer-diff-scope whole-PR (verdict notes
        |                                              PR-level imprecision)
        +---- timeout / hung ---------------------> BLOCKED + stderr surfaced
        +---- stale-sidecar (stop rule 7)        ---> HOLD route /safer:contract
        v
  Phase 5 verdict + Phase 6 publish (table rows for exits 10/11/12/13)
```

Exit-code routing is the mechanical surface of Principle 8 (the ratchet); no downstream modality may sidestep by editing the sidecar JSON or `@spec.*` directives directly to "make the error go away" (Invariant 2).

## 5. Errors

Every interface declares its error channel as a named tagged-error set. All errors below are discriminated by their `_tag` field and exposed via the `safer-spec validate --json` schema or the `bin/safer-gen-skills` renderer.

### 5.1 `bin/safer-gen-skills` (renderer + check)

```
type GenSkillsError =
  | { _tag: "MissingPrinciplesError";        path: string }
  | { _tag: "MissingVendorSkillError";       slug: string; resolvedPath: string }
  | { _tag: "StaleRenderError";              path: string; reason: "tmpl-newer" | "principles-newer" | "vendor-newer" }
  | { _tag: "SentinelUnsubstitutedError";    occurrences: ReadonlyArray<{ path: string; line: number }> }
```

The error channel surfaces as non-zero exit + STDERR string; `--check` returns first-error-only or all-errors per the existing `--check` semantics.

### 5.2 `safer-spec validate` (consumed by `/safer:verify` Phase 3)

The sister codemod owns these error tags; the plugin consumes them at the exit-code boundary. They are listed here for completeness — the plugin does not raise them, only routes on them.

```
type ValidateExitCode =
  | { _tag: "VersionSkewError";              code: 10; doctorOutput: string }
  | { _tag: "MissingSpecPropertyError";      code: 11; symbol: string; sidecarPath: string; recommendedRoute?: string }
  | { _tag: "MissingStubError";              code: 12; symbol: string; sidecarPath: string; recommendedRoute?: string }
  | { _tag: "MissingImplError";              code: 13; symbol: string; sidecarPath: string; recommendedRoute?: string }
```

The `recommendedRoute` field is the optional `safer-spec validate --json` override; spec Assumption 8 says its absence is acceptable for v0.2.0 with the `bin/safer-diff-scope` fallback.

### 5.3 `/safer:verify` Phase 3 timeout wrapper

```
type ValidateTimeoutError =
  | { _tag: "ValidateTimeoutError";          timeoutMs: 60000; pid: number; stderr: string }
```

The wrapper is a single-line `node -e` using `child_process.spawn` + `setTimeout`. The error surfaces as BLOCKED with the stderr verbatim in the verdict body. A unit test against a deliberately hung subprocess confirms the wrapper (spec acceptance bullet 21).

### 5.4 `/safer:setup` Step 4c

```
type SetupStep4cError =
  | { _tag: "NotTypeScriptError";            reason: "no-tsconfig" | "no-vitest" }
  | { _tag: "PluginRootUnresolved";          candidates: ReadonlyArray<string> }
  | { _tag: "VendorSubmoduleAbsent";         path: string }
  | { _tag: "CodemodInstallFailed";          $PM: string; stderr: string }
  | { _tag: "DoctorFailed";                  stderr: string }
```

Each error maps to a halt-with-pointer receipt; non-TS halts with the "use safer-by-default 0.1.x" pointer per spec Invariant 1.

The error union is exhaustive: every public function downstream `implement-*` writes will discriminate on the tag set with a `default: absurd(x)` branch (Principle 4).

## 8. Open questions

Architect-surfaced open questions, each with a recommended default and an escalation target. These are not blockers — defaults are picked unless the implementer or reviewer disagrees.

1. **Q (A1):** Should `bin/safer-gen-skills` resolve `{{> vendor-skill:<slug>}}` against a fixed prefix (`vendor/safer-spec-development/skills/<slug>/SKILL.md`) or accept an arbitrary repo-relative path?
   **Recommended default:** Fixed prefix. The two consumers (`contract-init`, `contract-migrate`) are the only vendor-skill surfaces today; a third would extend the design via amendment. The narrow form keeps the templating dialect tight. **Escalation target:** `/safer:contract`.

2. **Q (A2):** How does `$SAFER_SBD_PLUGIN_ROOT` resolve in `/safer:setup` Step 4c?
   **Recommended default:** Four-step lookup, in order: (a) `$CLAUDE_PLUGIN_ROOT` env var (set by the Claude Code plugin host), (b) `~/.claude/plugins/cache/safer-by-default/safer/<v>/` (the documented Claude Code cache path per `CLAUDE.md` "Where things live at runtime"), (c) `~/.local/share/safer-by-default/` (the documented `setup-codex` install path per `CLAUDE.md`), (d) `$SAFER_SOURCE_DIR` (source-dev fallback used by `setup-codex` and from-source development). First match wins. Halt with `PluginRootUnresolved` if none resolves. The four-step lookup covers every documented install posture. **Escalation target:** `/safer:contract`.

3. **Q (A3):** Submodule SHA pinning timing.
   **Recommended default:** Implementer pins at S1's merge SHA, not architect. Architect commits `.gitmodules`-equivalent design content (URL, path, intended pin); implementer runs `git submodule add` once S1 lands. This avoids a stale pointer if S1's branch rebases. **Escalation target:** `/safer:implement-staff`.

4. **Q (A4):** Should `bin/safer-gen-skills --check`'s sentinel arm run unconditionally or only in release-mode CI?
   **Recommended default:** Unconditionally. Sentinel-present in a committed template is a development-time issue (the integration PR hasn't substituted yet) — surfacing it earlier than release CI improves the loop. The development branch carries the sentinel intentionally; release CI gates on substitution. The contradiction resolves at the integration PR's substitution step. **Escalation target:** `/safer:contract` (if the trade-off needs adjudication).

5. **Q (A5):** Should the new `safer:contract` GitHub label be created at the plugin's own repo, or only at adopter repos?
   **Recommended default:** Both, but via the same script. `bin/safer-setup-labels` runs at adopter setup and creates the labels there. This plugin's own repo runs the same script as part of orchestrate's setup; the v0.2.0 implementer runs it once on `chughtapan/safer-by-default` to add `safer:contract`. **Escalation target:** `/safer:contract`.

These five sit alongside spec #304's four open questions (Q1-Q4 in spec body), which are unchanged by this design.

## 9. Submodule mechanism

This section answers the four mechanism questions in sub-issue #306 acceptance bullet 2.

### 9.1 Submodule path

**Canonical path:** `vendor/safer-spec-development/`

**Why this name:** matches spec #304 acceptance bullet 7's existing language ("a `vendor/safer-spec-development/skills/<name>/SKILL.md` snapshot ... committed to this repo"). Tree-shaped peer under `vendor/`. The folder name mirrors the sister npm package name (`@chughtapan/safer-spec-development`) so a reader scanning the repo can tell which upstream this points at without opening `.gitmodules`. `external/`, `deps/`, and `third_party/` were considered; `vendor/` wins because the spec text already says so and the `feedback_stop_over_engineering` memory rejects parallel naming.

**Setup command (run once by the implementer after S1 merges):**

```bash
git submodule add \
  https://github.com/chughtapan/safer-spec-development.git \
  vendor/safer-spec-development
cd vendor/safer-spec-development
git checkout <sister v0.2.0 release SHA>
cd ../..
git add .gitmodules vendor/safer-spec-development
git commit -m "vendor: pin safer-spec-development at v0.2.0 release SHA"
```

**Update mechanism (per sister release):**

```bash
cd vendor/safer-spec-development
git fetch && git checkout <new pinned SHA>
cd ../..
bin/safer-gen-skills            # re-render contract-init/-migrate
git add vendor/safer-spec-development skills/contract-init/SKILL.md skills/contract-migrate/SKILL.md
git commit -m "vendor: bump safer-spec-development to <SHA short>"
```

Per Non-goal 8: manual git commit on every bump. No `postinstall` hook. No CI auto-bump. No `bin/safer-vendor-skills` helper.

### 9.2 `bin/safer-gen-skills` reads wrapper bodies from the submodule

The current renderer uses awk to expand one directive (`{{> principles}}`). The extension adds a second directive (`{{> vendor-skill:<slug>}}`) that resolves to the submodule path.

**Path resolution:**

```
directive   : {{> vendor-skill:safer-spec-init}}
resolves to : $PLUGIN_DIR/vendor/safer-spec-development/skills/safer-spec-init/SKILL.md
```

Where `$PLUGIN_DIR` is already declared at the top of `bin/safer-gen-skills` as `$(cd "$(dirname "$0")/.." && pwd)`. No new env var.

**Awk template (added next to the existing `{{> principles}}` arm):**

```awk
/^\{\{> vendor-skill:[a-z0-9-]+\}\}$/ {
  # Extract slug between "vendor-skill:" and "}}"
  slug = $0
  sub(/^\{\{> vendor-skill:/, "", slug)
  sub(/\}\}$/, "", slug)
  vendor_path = vendor_root "/" slug "/SKILL.md"
  while ((getline line < vendor_path) > 0) print line
  close(vendor_path)
  next
}
```

`vendor_root` is passed in as a new `-v` flag from the bash wrapper: `-v vendor_root="$PLUGIN_DIR/vendor/safer-spec-development/skills"`. Mirrors the existing `-v principles_file="$PRINCIPLES_FILE"` flag.

**Missing path:** the awk `getline` returns 0 if the file is missing, producing empty output. To make the failure loud, the bash wrapper pre-validates every `{{> vendor-skill:<slug>}}` directive by extracting the slug and checking the file exists before rendering. Missing → exit 1 with `ERROR: vendor-skill <slug> not found at <path>`. This mirrors the existing `[ -f "$PRINCIPLES_FILE" ]` check.

**Generated SKILL.md output:** the wrapper file in the repo is one frontmatter block + one heading + one directive. The generated `SKILL.md` is the vendor body inlined verbatim, with the AUTO-GENERATED marker stamped after the frontmatter (existing mechanism). The reader of `skills/contract-init/SKILL.md` sees the sister-package skill body unchanged from the upstream snapshot.

### 9.3 `/safer:setup` resolves `safer-spec` through the submodule

**Install command (in `skills/setup/SKILL.tmpl` Step 4c):**

```bash
SAFER_SBD_PLUGIN_ROOT="${CLAUDE_PLUGIN_ROOT:-}"
if [ -z "$SAFER_SBD_PLUGIN_ROOT" ]; then
  # Fallback (a): documented Claude Code cache path per CLAUDE.md.
  SAFER_SBD_PLUGIN_ROOT=$(ls -dt ~/.claude/plugins/cache/safer-by-default/safer/*/ 2>/dev/null | head -1)
fi
if [ -z "$SAFER_SBD_PLUGIN_ROOT" ] || [ ! -d "$SAFER_SBD_PLUGIN_ROOT/vendor/safer-spec-development" ]; then
  # Fallback (b): setup-codex install path per CLAUDE.md.
  if [ -d "$HOME/.local/share/safer-by-default/vendor/safer-spec-development" ]; then
    SAFER_SBD_PLUGIN_ROOT="$HOME/.local/share/safer-by-default"
  fi
fi
if [ -z "$SAFER_SBD_PLUGIN_ROOT" ] || [ ! -d "$SAFER_SBD_PLUGIN_ROOT/vendor/safer-spec-development" ]; then
  # Fallback (c): SAFER_SOURCE_DIR (development-from-source path).
  if [ -n "${SAFER_SOURCE_DIR:-}" ] && [ -d "$SAFER_SOURCE_DIR/vendor/safer-spec-development" ]; then
    SAFER_SBD_PLUGIN_ROOT="$SAFER_SOURCE_DIR"
  fi
fi
if [ -z "$SAFER_SBD_PLUGIN_ROOT" ] || [ ! -d "$SAFER_SBD_PLUGIN_ROOT/vendor/safer-spec-development" ]; then
  echo "ERROR: cannot locate safer-by-default plugin root with vendor/safer-spec-development/" >&2
  echo "       Re-run /plugin install safer@safer-by-default (Claude Code) or ./setup-codex (Codex)," >&2
  echo "       then retry /safer:setup." >&2
  exit 1
fi
$PM add -D "file:$SAFER_SBD_PLUGIN_ROOT/vendor/safer-spec-development"
```

The three-step lookup covers the three documented install paths (Claude Code marketplace cache, Codex symlink target, source clone). Order is: env-var first (fastest), then Claude Code cache (most common), then Codex path, then source-dev fallback.

**Why a `file:` install and not the npm name:** parent epic revision 2 defers npm publish; the sister package is not yet on the registry. The `file:` URL points the adopter's package manager at the submodule's working-tree path inside the plugin cache. `pnpm`, `npm`, and `yarn` all support `file:` URLs; `bun` supports them via `bun add <path>` syntax. Step 4c branches on `$PM` for the syntax form (one `case` statement, three lines per arm).

**Post-publish migration:** when the sister package ships to npm (publish follow-up), Step 4c's install command swaps from `file:...` to `@chughtapan/safer-spec-development@~X.Y.Z`. The `__SAFER_SPEC_VERSION__` sentinel substitution mechanism (spec Invariant 12) is ready for that swap: the SKILL.tmpl source carries the sentinel even in v0.2.0; the release-PR substitution step replaces it with the actual `~X.Y.Z` pin at marketplace publish time. For v0.2.0, the substituted value is the `file:` form; v0.2.x post-publish, the substituted value is the npm form.

### 9.4 `/safer:verify` resolves the gate

**Phase 2 detection probe (in `skills/verify/SKILL.tmpl`):**

```bash
if ! $PM exec safer-spec doctor >/tmp/safer-verify-$PR/doctor.log 2>&1; then
  # BLOCKED — doctor failed
  VERDICT=BLOCKED
  cat /tmp/safer-verify-$PR/doctor.log  # surface verbatim in verdict body
  exit 0
fi
```

**Phase 3 ring-1 gate (added to the existing `$LINT_CMD`/`$TYPECHECK_CMD`/`$TEST_CMD` block):**

```bash
SAFER_VALIDATE_TIMEOUT_MS="${SAFER_VALIDATE_TIMEOUT_MS:-60000}"
node -e '
  const { spawn } = require("child_process");
  const [, , pm, ...rest] = process.argv;
  const timeoutMs = Number(process.env.TIMEOUT_MS) || 60000;
  const child = spawn(pm, rest, { stdio: "inherit" });
  const timer = setTimeout(() => { child.kill("SIGKILL"); process.exit(124); }, timeoutMs);
  child.on("exit", (code) => { clearTimeout(timer); process.exit(code ?? 1); });
' -- "$PM" exec safer-spec validate --implemented \
  > /tmp/safer-verify-$PR/spec-validate.log 2>&1
SPEC_VALIDATE_EXIT=$?
```

The wrapper passes `$PM` and its argv through Node's `process.argv` rather than shell-interpolating into the `-e` string. This eliminates the quoting risk if `$PM` ever contains a space or shell metacharacter, and keeps the timeout helper a single self-contained one-liner per spec Invariant 4 (no new `bin/` helper). Timeout default is 60s, overridable per-invocation via `SAFER_VALIDATE_TIMEOUT_MS` (env var read from the calling shell, forwarded into Node via `TIMEOUT_MS=$SAFER_VALIDATE_TIMEOUT_MS` in the actual SKILL.tmpl emission — abbreviated here for readability).

Exit codes map per §4.3. `124` is the conventional timeout exit; verify surfaces it as BLOCKED with stderr.

`$PM exec` resolves the codemod via the ADOPTER project's `node_modules/.bin` (the install happened during `/safer:setup`). The submodule is not directly referenced at verify time; setup's `file:` install put the codemod in `node_modules` and `$PM exec` finds it there. This matches spec Invariant 12 (a): local-first binary resolution via `$PM exec`, never `command -v`.
